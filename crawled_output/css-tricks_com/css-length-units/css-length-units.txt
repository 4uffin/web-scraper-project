Overview

Whatever they are, the unit that follows a number determines the number‚Äôs computed length. And by ‚Äúlength‚Äù we mean any sort of distance that can be described as a number, such as the physical dimensions of an element, a measure of time, geometric angles‚Ä¶ all kinds of things!
At the time of this writing, the CSS Values and Units Module Level 4 specification defines a bunch of different CSS units ‚Äî and many of those are relatively new (this pun will make sense later).

Types of numbers
You may think a number is just a number, and you‚Äôre not wrong. Numbers are numbers! But we can distinguish between a few different types of numbers, which is helpful context for discussing the different types of units we attach them to since ‚Äúnumber‚Äù can mean, well, a number of different things.
Integers (literally a unit-less number, e.g. 3)
Numbers (same as an integer, only measured in decimals, e.g. 3.2)
Dimensions (either a number or integer with a unit, e.g. 3.2rem)
Ratios (the quotient between two divided numbers, e.g. 3/2)
Percentages (e.g. 3%)
Got that? They‚Äôre all numbers but with nuances that make them ever-so-slightly different.
From here, we can think of numbers in CSS as falling into two specific types of units:¬†absolute¬†and¬†relative. Let‚Äôs start things off our deep dive on CSS length units by breaking those down.
Absolute units
An¬†absolute unit¬†is like Bill Murray in the movie¬†Groundhog Day: it‚Äôs always the same. In other words, whatever the number is, that‚Äôs exactly how it computes in the browser regardless of how other elements are sized.
The most common absolute value you‚Äôll see is the¬†pixel¬†value. It‚Äôs sort of hard to define, but a pixel is the smallest building block of a graphical display, like a computer screen. And it‚Äôs based on the resolution of the screen. So, if you‚Äôre on a super high-resolution screen, a pixel will be smaller than it would be on a low-resolution screen, as the resolution can pack more pixels into a smaller amount of space for higher clarity. Absolute values are nice in that they are predictable. That could, however, change in some situations, particularly when it comes to zooming. Relative units
A¬†relative unit¬†is extremely well-named because whatever value we use for a relative unit depends on the size of something else. Say we have an HTML element, a¬†<div>, and we give it an absolute height value (not a relative one) of¬†200px.
<div class="box">
I am 200 pixels tall
</div>
.box {

}
That height will never change. The¬†.box¬†element will be¬†200px¬†tall no matter what. But let‚Äôs say we give the element a relative width (not an absolute one) of¬†50%.
<div class="box">
I am 200 pixels tall and 50% wide
</div>

}
What happens to our box? It takes up 50%, or half, of the available space on the screen.

Percentages, like 50%, are only one kind of relative unit. We have many, many others. Once again, it‚Äôs helpful to break things out into separate groups to understand the differences just as we did earlier with absolute units.
Percentages
We‚Äôve already discussed percentages in pretty good detail. What makes a percentage relative is that it computes to a number value based on the length of another element. So, an element that is given width: 25% in a container that is set to width: 1000px computes to width: 250px.

There‚Äôs something else that‚Äôs very important to know when working with viewport units. You know how mobile phones, like iPhone or an Android device, have virtual keyboards where you type directly on the screen? That keyboard changes the size of the viewport. That means that whenever the keyboard opens, 100vh is no longer the full height of the screen but whatever space is leftover while the keyboard is open, and the layout could get super squished as a result.
That‚Äôs why we have the svh, lvh, and dvh units in addition to vh:
svh is equal to the ‚Äúsmall‚Äù viewport height, which occurs when the keyboard is open.
lvh is equal to the ‚Äúlarge‚Äù viewport height, which is when the keyboard is disabled and out of view.
dvh is a happy medium between svh and lvh in that it is ‚Äúdynamic‚Äù and updates its value accordingly when the keyboard is displayed or not.
dvmin / dvmax is the greater ore lesser of dvh, respectively.

What about unit-less numbers?
Oh yeah, there are times when you‚Äôre going to see numbers in CSS that don‚Äôt have any unit at all ‚Äî just a single integer or number without anything appended to it.

This isn‚Äôt a comprehensive list of all the CSS properties that accept unit-less numeric values, but it is a solid picture of when you would use them. You‚Äôll see that in most cases a unit-less number is an explicit detail, such as a specific column to position an element, a specific layer in a stacking context, a boolean that enables or disables a feature, or the order of elements. But note that anytime we declare ‚Äúzero‚Äù as a number, we can write it with or without a prepended unit, as zero always evaluates to zero no matter what unit we‚Äôre dealing with.

We can create our own custom units!
In some cases, we may want to work with a numeric value, but CSS doesn‚Äôt exactly recognize it as one. In these cases, the number is recognized as a ‚Äústring‚Äù value instead, regardless of whether or not it contains alphabetical characters. That‚Äôs where we can use @property to create what‚Äôs called a ‚Äúcustom property‚Äù that evaluates a numeric value in a certain way.
Here‚Äôs a good example. There was a time when it was virtually impossible to animate a gradient that changes colors over time because to do so would require (1) a color function that allows us to change a color value‚Äôs hue (which we have with hsl()) and (2) being able to interpolate that hue value around the color spectrum, between a range of 0deg and 360deg.
Sounds simple enough, right? Define a variable that starts at 0 and then cycles through 360 degrees at the end of an animation. But this doesn‚Äôt work:
/* üëé */
.element {
--hue: 0;
animation: rainbow 10s linear infinite;
background: linear-gradient(hsl(--hue 50% 50%);
}
@keyframes rainbow {
from { --hue: 0; }
to { --hue: 360deg; }
}
That‚Äôs because CSS reads the variable as a string instead of a number. We have to register that variable as a custom property so that CSS aptly reads it as a numeric value.
@property --hue {
syntax: "<number>";
initial-value: 0;
inherits: true;
}
There we go! Now that we‚Äôve given CSS a hint that the --hue syntax is that of a <number>, we can animate away!
/* üëç */
@property --hue {
syntax: "<number>";
initial-value: 0;
inherits: true;
}
.element {
--hue: 0;
animation: rainbow 10s linear infinite;
background: linear-gradient(hsl(--hue 50% 50%);
}
@keyframes rainbow {
from { --huw: 0; }
to { --hue: 360deg; }
}

When to use one unit over another
This is super tricky because CSS is extremely flexible and there are no definitive or hard-and-fast rules for when to use a specific type of CSS length unit over another. In some cases, you absolutely have to use a specific unit because that‚Äôs how a certain CSS feature is specced, like using angle units to set the direction of a linear gradient:
.element {
background: linear-gradient(
135deg, red, blue;
)
}
The same goes for the values we use in certain color functions, like using percentages to set the saturation and lightness levels in the hsl() function:
.element {
background: hsl(0 100% 10%);
}
Speaking of color functions, we define alpha transparency with either an integer or number:
.element {
background: hsl(0 100% 10% / 0.5); /* or simply .5 */
}
All that being said, many cases are going to be some degree of ‚Äúit depends‚Äù but there are some instances where it makes sense to use a specific unit in your work.
Generally set font sizes in rem and em units
This way, you can set things up in a way where changing the font-size value of the <html> or a parent element updates the font sizes of their descendants.
html {
font-size: 18px; /* Inherited by all other elements */
}
.parent {
font-size: 1rem; /* Updates when the `html` size changes */
}
.child {
font-size: 1em; /* Updates when the parent size changes */
}
Declare ‚Äúzero‚Äù without units if you‚Äôd like
It‚Äôs not a big deal or anything, but leaving off the units shortens the code a smidge, and anytime we‚Äôre able to write shorter code it‚Äôs an opportunity for faster page performance. The impact may be negligible, but we‚Äôre able to do it since 0 always computes to 0, no matter what unit we‚Äôre working with.
Use container units for responsive design, where possible
Container queries in general are so gosh-darn great for responsive layouts because they look at the size of the container and let us apply styles to its descendants when the container is a certain size.
.parent {
container: my-container / inline-size; /* Looks at width */
}
.child {
display: flex;
flex-direction: column;
max-width: 100vh; /* 100% of the viewport */
}
/* When the container is greater than 600px wide */
@container my-container (width >= 600px) {
.child {
flex-direction: row;
max-width: 50%; /* 50% of the parent elenent */
}
}
So, if we going to size the .child element ‚Äî or any of its children ‚Äî it‚Äôs worth specifying sizes in relation to the container‚Äôs size with container units than, say, the viewport‚Äôs size with viewport units.
.parent {
container: my-container / inline-size; /* Looks at width */
}
.child {
display: flex;
flex-direction: column;
max-width: 100cqi; /* 100% of the container */
}
/* When the container is greater than 600px wide */
@container my-container (width >= 600px) {
.child {
flex-direction: row;

}
}
Use percentages when you‚Äôre unsure of the context
Yes, use container units for responsive design, but that only does you good if you know you are in the context of a container. It‚Äôs possible, though, that you use the same component in different places, and one of those places might not be a registered container.
In that case, go with a percentage value because percentages are relative to whatever parent element you‚Äôre in, regardless of whether or not it‚Äôs a container. This way, you can declare an element‚Äôs size as 100% to take up the full space of whatever parent element contains it.
The only word of caution is to note that we‚Äôre only basing the size on the parent. Meanwhile, container units can style any descendant in the container, no matter where it is located.
Viewport units are great for laying out containers
You may be thinking that viewport units are a bad thing since we‚Äôve been advising against them in so many cases, like font sizing, but they are still incredibly useful, particularly when it comes to establishing a full-page layout.
I say ‚Äúfull-page‚Äù layout because container queries are the gold standard for sizing elements according to the space they have in their container. But if we‚Äôre working with a full page of containers, this is where viewport units can be used to establish a responsive layout at a higher level.
