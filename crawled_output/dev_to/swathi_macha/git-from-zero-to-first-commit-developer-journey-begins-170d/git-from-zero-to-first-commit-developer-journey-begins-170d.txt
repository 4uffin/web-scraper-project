Step 2: Tell Git Who You Are
Every time you commit, Git saves your name and email. This helps teams know who made what change.

First, what is git config?
git ‚Üí is the tool we‚Äôre using (the version control system).
config ‚Üí stands for configuration ‚Üí basically ‚Äúsettings‚Äù for Git.
So, git config = ‚ÄúHey Git, I want to set or change your settings.‚Äù
Check settings:
--global tells Git:
‚ÄúApply this setting everywhere on this computer, for all my Git projects.‚Äù
Without --global, Git would only apply the setting to one specific project.
Example:
If Ravi works on 5 different projects on his laptop, setting --global means Git knows who he is in all those projects, not just one.
What are user.name and user.email?
Every time you commit (save a snapshot in Git), Git records:
‚Ä¢ Who made the change (user.name)
‚Ä¢ Their email address (user.email)
So in the future, anyone looking at the project can see:

Date:
Tue Sep 17 2025
Message: Added Forgot Password button
This is like signing your name when you make an edit.
git config --list

Exit fullscreen mode
This command asks Git: ‚ÄúShow me all the settings (configuration) you currently know about.‚Äù
In real teams, this is critical. If Ravi breaks the login, everyone knows who to call.
Step 3: Create a Project and Initialize Git
Let‚Äôs say your team is starting the ‚Äúbanking-web‚Äù service.

Enter fullscreen mode
Exit fullscreen mode
mkdir ‚Üí Stands for make directory.
A directory is just another word for a folder.
So this command means: ‚ÄúCreate a new folder called banking-web here.‚Äù
Think of it like going to your desktop, right-clicking, and choosing:
New Folder ‚Üí Name it banking-web
cd ‚Üí Stands for change directory.
It tells the terminal: ‚ÄúGo inside the folder I just created.‚Äù
So now, Ravi‚Äôs ‚Äúcurrent location‚Äù is inside the banking-web folder.
(Just like double-clicking the folder in File Explorer/Finder to open it).
&&
This is a little trick:
‚Ä¢ && means ‚Üí ‚ÄúRun the next command only if the previous one worked successfully.‚Äù
So here:
1.
Make the folder (mkdir banking-web)
2.
If that succeeds, go inside it (cd banking-web).
All in one line.
git init
Now the magic:
git init ‚Üí Initializes a brand-new Git repository inside the current folder.
It creates a hidden folder called .git/ inside banking-web.
That .git/ folder stores:
Commit history
Branches
Configurations
Basically, the brain of Git for this project.
Without git init, your folder is just a normal folder.
With git init, your folder is now tracked by Git ‚Üí every change can be saved, reverted, or collaborated on.
Putting it all together
With just these two lines:
mkdir banking-web && cd banking-web
git init
We are saying:
1.
Create a new project folder called banking-web.
2.
Go inside it.
3.
Turn it into a Git project (so changes are tracked).
So, in simple words:
mkdir + cd = Create a workspace.
git init = Tell Git, ‚ÄúHey, start tracking everything I do in this workspace.‚Äù
Step 4: Create a File
You create your first file:

Enter fullscreen mode
Exit fullscreen mode
echo is a command that simply prints text to the screen.
So here, echo "# Banking Web App" means: Print the text # Banking Web App.
( > Redirection operator)
Normally, echo just shows text on the screen. But the > symbol redirects the output into a file.
">" means ‚Üí ‚ÄúDon‚Äôt show it on the screen. Instead, put it inside this file.‚Äù
If the file doesn‚Äôt exist ‚Üí it creates it. If the file already exists ‚Üí it overwrites it.
README.md is the first file people usually see in a project.
.md means Markdown file ‚Äî a lightweight format for writing text with headings, bold, lists, etc.
The # in Markdown means ‚Üí Heading 1. So # Banking Web App will render as a big heading: Banking Web App
We are saying:
1.
Use echo to print the text # Banking Web App.
2.
Redirect (>) that text into a new file called README.md.
So after running this command:
‚Ä¢ A file named README.md will be created in the folder.
‚Ä¢ Its contents will be:
So, this command creates a new README file with the heading: Banking Web App
Now check what Git thinks:

Exit fullscreen mode

Exit fullscreen mode
Why ‚ÄúUntracked‚Äù?
Because Git sees the file exists but isn‚Äôt tracking it yet.
Untracked = new file not added to Git.
Step 5: Stage the File
You decide: ‚ÄúYes, I want to save README.md in Git.‚Äù
So you stage it:
git add README.md
Enter fullscreen mode
Exit fullscreen mode
It means: ‚ÄúHey Git, please include this file in the next save (commit)
Now check again:

Enter fullscreen mode
Exit fullscreen mode

The staging area is like a shopping basket ‚Äî you select exactly what changes to save in the next commit.
Step 6: Commit the File
Now you save this snapshot in Git‚Äôs history:
git commit -m "Initial commit: add README"
Enter fullscreen mode
Exit fullscreen mode
git commit ‚Üí tells Git: ‚ÄúSave a snapshot of everything in the staging area.‚Äù
-m ‚Üí means ‚Äúmessage‚Äù.
Instead of opening an editor, you give the commit a description right here in quotes.
"Initial commit: add README" ‚Üí your description.
This is what shows up in the history when people look at the project later.
In a real company, every commit tells a story:
‚Ä¢ Who made it.
‚Ä¢ What changed.
‚Ä¢ When it happened.
Step 7: See Your Commit
Check the history:

git log = shows the history of commits (snapshots) in your project.
--oneline flag makes the history short and simple:
‚Ä¢Only shows:
‚Ä¢ Short commit ID (first 7 characters of the hash)
‚Ä¢ Commit message

Step 9: Understand File States in Git
Here‚Äôs the file lifecycle you just experienced:
1.
Untracked ‚Üí file exists, not added to Git.
2.
Staged ‚Üí selected for commit (git add).
3.
Committed ‚Üí permanently saved in history (git commit).
4.
Modified ‚Üí file was edited after commit but not yet staged.
Check with:

Step 10: Ignore Junk Files
In real IT projects, you don‚Äôt want logs, caches, or personal notes in Git.
Create .gitignore:

Step 11: Local vs Remote
Right now, everything is local ( i.e exists only on your laptop ). But teams work together using a remote repository (GitHub, GitLab, Bitbucket).
Solution ‚Üí Remote repository. This is like a shared cloud backup on GitHub/GitLab.
Connect to remote:
git remote add origin https://git hub.com/acme/banking-web.git
git branch -M master
git push -u origin master
Enter fullscreen mode
Exit fullscreen mode
git remote add origin https: //git hub.com// .. //banking-web.git
git remote add ‚Üí means ‚Äúconnect my local repo to another repo on a server‚Äù.
origin ‚Üí is the nickname (alias) for the remote repo.
By convention, the main remote is always called origin.
https: //git hub.com// .. //banking-web.git ‚Üí the URL of the GitHub repository.
Think of it like: ‚ÄúTell my laptop: this project also lives on GitHub at this address.‚Äù
git branch -M master
git branch -M ‚Üí renames the current branch (forcefully if needed).
By default, your branch might be called main (new Git versions) or master (older versions).
This makes sure both local and remote use the same branch name.
git push -u origin master
git push ‚Üí upload commits from your local repo ‚Üí to the remote repo.
origin ‚Üí the remote repo nickname.
master ‚Üí the branch you are pushing.
-u (set upstream) ‚Üí tells Git:
‚ÄúFrom now on, when I type git push or git pull without extra arguments, use origin/master by default.‚Äù
This is the first time you‚Äôre sending your project to GitHub.
Now Ravi, Priya, and Neha can git clone and work together.
The repo now lives on both your laptop and GitHub.
Your teammates don‚Äôt need your laptop ‚Äî they can just run:
git clone https://github.com/acme/banking-web.git
Now they have their own local copy of the repo.
Your Git Journey Today
‚Ä¢ git init ‚Üí Start a repo.
‚Ä¢ git status ‚Üí See what‚Äôs happening.
‚Ä¢ git add ‚Üí Stage files.
‚Ä¢ git commit -m "msg" ‚Üí Save snapshot.
‚Ä¢ .gitignore ‚Üí Keep junk out.
‚Ä¢ git log --oneline ‚Üí Review history.
‚Ä¢ git push ‚Üí Share with team.
Example
Each dev creates commits, pushes to the remote, and pulls each other‚Äôs changes.
Without Git, files would get overwritten. With Git, everything is tracked, reversible, and collaborative.
NOTE:
Think of Git as having three areas:
Working Directory
Where you actually create or edit files. (e.g., editing README.md in VS Code)
Staging Area (Index)
A basket where you collect files you want to save in the next snapshot.
Repository (Commits)
The permanent history of your project (inside .git/).
Conclusion
Git isn‚Äôt just a tool, it‚Äôs your time machine and safety net. Every change is saved, nothing is lost, and teamwork becomes smooth.
On day one at your job, if you know just these commands, you‚Äôll already feel like part of the dev team.
