When Dinosaurs Survived the Meteor: From Python to Deno
#deno
#typescript
#javascript
#python
I started seriously learning Python in 2020 while working on a release note generator for a co-op bank. It was easy to understand, enjoyable to work with, and a perfect excuse to learn something new. If you‚Äôre anything like me, you love experimenting, trying new things, and continuously improving.
From that moment, Python became my default language for almost everything. I built REST APIs with FastAPI and Flask, analyzed data with Jupyter, created CLI apps with Click‚Äîyou name it. I practically ate Python for breakfast.
Funnily enough, at the time I was using TypeScript and Node.js exclusively. It wasn‚Äôt perfect for my DevOps work, but it got the job done.
Then one day, I decided to fully dive into Python. I didn‚Äôt know much about it at first, but I started learning while building the release note generator‚Äîand from that point, I was hooked.
So, what happened? Why the change then?
To be clear, I‚Äôm not saying Python is a bad language. I‚Äôll still use it for specific cases‚Äîit‚Äôs a great tool for quickly building software.
The issues I ran into weren‚Äôt always about Python itself. Some came from the ecosystem around it, while others stemmed from the burnout I experienced trying to push Python into tasks it wasn‚Äôt originally designed for. Over time, that friction started to outweigh the joy of using the language.
Batteries not included
Python‚Äôs package management‚Ä¶ well, it‚Äôs a bit of a mess. By default, you get pip. Install a package, and it‚Äôs installed system-wide without generating a dependency file like Node‚Äôs package.json. Want to run your app on another environment with all the right packages? Good luck‚Äîyou need to remember every package and its version. Sure, pip freeze > requirements.txt exists, but it‚Äôs not automatic. Every time you add a new package, you have to run it again. And don‚Äôt even think about forgetting to activate a virtual environment‚Äîrun it globally, and you‚Äôll dump your entire system‚Äôs Python packages into requirements.txt.
You could use Poetry, pyenv, or other tools, but when I start a new project, I expect the language to come with a built-in, usable package manager. I don‚Äôt want to bolt on another tool just to get basic functionality that other ecosystems provide out of the box.
Running your app in Python isn‚Äôt much better. In Node.js, you have npm run start; in Spring, mvn spring-boot:run. Python? You either hope someone documented it in the README or added a Makefile. Neither is native, straightforward, or consistent.
And virtual environments‚Ä¶ don‚Äôt get me started. They‚Äôre relics of the past, adding more friction than they solve. Clunky activation scripts behave differently across shells and OSes, manual management is error-prone, and reproducibility is weak. In today‚Äôs world‚Äîwhere containers, Poetry, and even Python‚Äôs built-in venv exist‚Äîvirtualenvs feel outdated, brittle, and inefficient. They make what should be simple into a chore.
Fitting a cube in the triangle (or How I Got Burnt Out)
You know that classic toy with differently shaped holes and matching blocks? Ever tried to force a cube into the triangle hole? Of course, it doesn‚Äôt fit. The obvious choice is the square hole, designed perfectly for the cube. But if you‚Äôre stubborn like me, you might twist, turn, even grab a saw to reshape the triangle‚Äîonly to realize you‚Äôve essentially recreated the square hole. You could have just used the original square hole all along.
Where am I going with this? Programming languages are similar‚Äîeach shines in specific areas. Java excels at object-oriented programming, TypeScript keeps your types in check, and COBOL still rules the banking world with stable systems. My gripe isn‚Äôt with Python itself‚Äîit‚Äôs how it‚Äôs sometimes used.
Python is multi-paradigm: you can write object-oriented, procedural, functional, or hybrid code. But multi-paradigm doesn‚Äôt mean perfect at everything. I once worked on a REST API with FastAPI. Initially, it was simple and easy to follow. Then came the ‚Äúlet‚Äôs make it more object-oriented for maintainability‚Äù phase. Suddenly, every feature required complex design patterns, turning the code into spaghetti. Understanding it became a full-time job, and changes felt like climbing a mountain. It even contributed to burnout‚ÄîI started dreading Python and FastAPI altogether.
From that experience, I learned: complexity creeps in fast if your team doesn‚Äôt agree on core principles like KISS (Keep It Simple, Stupid) and DRY (Don‚Äôt Repeat Yourself). Focusing too much on ‚Äúperfect‚Äù code instead of delivering value is a trap. Clients care about results, not how beautifully your software is architected.
Moral of the story: Don‚Äôt force a cube into a triangle hole. Use the right tool for the job, and save yourself the headache.
Deno: From one reptile to another
You‚Äôve probably guessed it by now‚Äîgoing forward, I‚Äôll be using Deno with TypeScript for my projects. I was already familiar with TypeScript and Node.js, but Deno was new territory for me. So, what exactly is Deno? Deno is a modern JavaScript runtime created by Ryan Dahl, the same person who built Node.js.
Its main goal is simple: provide an uncomplicated way to develop software with JavaScript and TypeScript, without many of the headaches that come with Node.js.
I‚Äôve been working with Deno for a while now, and I‚Äôve become a huge fan. I won‚Äôt dive into every advantage Deno has over Node.js or Python‚Äîjust the ones that matter most to me.
TypeScript supported natively
No more tsconfig migraines or wasting time tweaking endless compiler options. Forget about juggling separate build and transpile steps, or wiring up an extra bundler just to get code running. And best of all, no more CommonJS vs ESM soap opera‚Äîyou can finally stop googling import/export errors at 2 a.m. and just write code that works.
Batteries included
Deno comes with a suite of built-in tools: a test runner, a linter, a formatter, and a package manager. That means you can skip the usual headache of configuring each tool separately. For me, the two that really stand out are the test runner and the package manager‚Äîthey make writing, running, and managing your code a breeze right out of the box.
Package manager
One of Deno‚Äôs standout features is its package manager. When you install a new package, Deno automatically adds the reference to your deno.json file‚Äîno need for a separate npm install when you move your project to another machine. Say goodbye to pip freeze chaos or juggling virtual environments just to get your app running.
Running your software is just as straightforward. Deno provides a clear and consistent way to execute your code, and you can even configure tasks directly in your deno.json file to simplify your workflow.
Here‚Äôs an example of a deno.json configuration:
{
"imports": {
"@hono/swagger-ui": "jsr:@hono/swagger-ui@^0.5.2",
"@hono/zod-openapi": "npm:@hono/zod-openapi@^1.1.0",
"hono": "jsr:@hono/hono@^4.9.5",
"hono-openapi": "npm:hono-openapi@^0.4.8",
"mongoose": "npm:mongoose@^8.0.0",
"zod": "npm:zod@^4.1.5"
},
"tasks": {
"start": "deno run --watch -A src/main.ts"
},

}
}

Test runner

// idOdd_test.ts
Deno.test("isEven", () => {

});
